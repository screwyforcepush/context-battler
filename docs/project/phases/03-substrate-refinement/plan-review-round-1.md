# Phase 03 — Plan Review Round 1

## Review Summary
- **Overall assessment:** **Concern**. The plan is aligned with the North Star, but it is not ready for WP-A/WP-B dispatch until the blocking contract gaps below are folded into the work packages.
- **What is solid:** Schema-first sequencing is the right posture for the POC break. The core target state matches `mental-model.md` §11: outcome attribution, wall visibility, drained-corpse traceability, structured overwatch, and raw replay introspection. WP-C and WP-D are genuinely disjoint at runtime: grep found no `apps/replay/src` imports from `convex/llm/*` and no `convex/llm/*` imports from replay code; the only renderer/backend coupling is generated Convex types.
- **What is risky or unclear:** WP-B currently cannot compile as sequenced, `_internal_runMatch.ts` is a missed schema mirror, the closing-10 report path does not yet have a data-flow decision for metrics that require turn-level `agentRecords`, and the concept-spec edit target is too narrow.

## Spec / Guide Deviations
- **AOP.CALIBRATE / source-of-truth alignment:** The phase docs read as aligned with `mental-model.md` §11, but `concept-spec.md` remains stale outside §11/§13/§21. That violates the repo documentation protocol that specs stay source-of-truth when the design diverges.
- **Architecture §1 boundary:** No runtime engine/renderer coupling was found. `apps/replay/src` imports generated Convex types/API only, not `convex/llm` or `convex/engine` runtime modules.
- **North Star scenario coverage:** Engine semantics are mostly covered in work-package tests, but raw prompt teaching and reasoning persistence need explicit unit-level checks. The offensive-overwatch scenario should be visible in README §3 even if WP-B already covers it.
- **Hard out-of-scope:** No WP scope item creeps into procedural maps, fog-of-war, public spectator/auth, cursed item content, live streaming, or migration shims. The copy-to-clipboard raw pane is within the personal overseer scope.

## Decision Notes
- **Dependency map:** `WP-A -> WP-B -> (WP-C || WP-D) -> WP-E` is directionally correct after moving `affordances.ts` deletion out of WP-B or making WP-B remove the import. WP-C and WP-D write sets are otherwise disjoint.
- **Trace kind for chest opens:** PM/architect must decide whether unified `action.kind="loot"` also means trace `resolution.actions[].kind="loot"` for chest opens. That decision controls `reconstruct.ts`, `HoverCard.tsx`, report metrics, and result-vocabulary tests.
- **Report aggregation architecture:** Decide whether phase-3 metrics live on per-run `runs` rows or are computed by a phase-3 report writer over raw turn ledgers. Current `reports.create` cannot compute them from existing `RunSummary`.
- **Reasoning Branch B:** If Azure exposes no reasoning text, `rationale` must be treated as the diagnostic substitute, not a nice-to-have optional sentence. The metric source and replay UI fallback should be symmetrical.
- **Approval gate:** Do not start WP-A until the schema mirror (`_internal_runMatch.ts`), reasoning nullability, trace-action attribution, and reporting data-flow decisions are reflected in the plan.
