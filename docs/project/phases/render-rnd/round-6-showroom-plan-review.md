# Round-6 Showroom Plan Review

Date: 2026-05-27
Spec: [`round-6-showroom-spec.md`](./round-6-showroom-spec.md)
Reviewer role: Review Architect

## Review Summary

- Overall assessment: **Concern - CHANGES_REQUESTED before WP-A implementation.**
- What is solid: the spec correctly frames Showroom as a curator diagnostic surface, keeps the work inside `throwaway-prototypes/d-full-match/`, preserves the no-UAT discipline, reuses `SceneBuilder`, `CameraRig`, and `EquipmentMeshAttachment`, and correctly requires the scale fix to affect both Showroom and replay through the shared manifest path.
- What is risky or unclear: the recommended factory extraction conflicts with the current ownership of `CHARACTER_MODEL_SCALE`, the AABB height assertion is treated as optional in one section despite being the core acceptance proof, fallback labels do not yet have enough structured animation-resolution state, the CameraRig "MODE_FREE only" claim is contradicted by existing input/control behavior, and the example showroom layout mixes raw world positions with a too-small SceneBuilder map.

## Spec / Guide Deviations

- **No Convex / HTTP / snapshot-schema scope creep found.** The plan keeps production code untouched and Showroom event synthesis local to the throwaway prototype. The synthetic map dictionary is acceptable as an internal `SceneBuilder` input, not a new persisted snapshot format.
- **No browser/UAT violation found in the spec.** The no-browsertools/no-screenshots boundary is explicit and repeated.
- **Renderer pathing boundary is preserved.** The plan does not introduce renderer pathfinding or geometry reasoning beyond using existing SceneBuilder/CameraRig surfaces.
- **Material pipeline alignment is good.** Armor remains material-change through `EquipmentMeshAttachment.update_equipment`, matching mental-model §13's material-swap rule.
- **Deviation requiring amendment:** Camera acceptance says `MODE_FREE` inspection, but current CameraRig controls can leave free mode unless disabled or locked for Showroom.

## Decision Notes

- Q1 `manifest.schemaVersion` 3 -> 4: **Ratify bump to 4.** This is throwaway-local, schema-like, and makes the manifest field addition explicit. Update `verify-scaffold.mjs` accordingly. This must not affect `MatchPlayer` snapshot `schemaVersion === 3`.
- Q2 factory vs pass-through: **Ratify factory extraction only after base-scale ownership is amended.** Pass-through is not preferred because WP-B would duplicate character-loading logic.
- Q3 tier-to-asset mapping: **Defer to engineer with constraints.** Use only existing manifest assets and document the selected mapping in code/readout. Recommended default: weapon low `rusty_blade`, mid `sword`, high `greatsword`; armor low `cloth`, mid `chain`, high `plate`.
- Q4 death loop override: **Amend.** Death should hold pose without mutating shared imported Animation resources unless the engineer proves the mutation is instance-local.
- Q5 row vs grid: **Ratify row.** Amend sizing so the row, labels, floor, wall, and cover share one coordinate system and fit default framing.
- Q6 gore exclusion: **Ratify no gore.** This matches the user's curation ask and avoids persistent blood clutter.
- Q7 fixed spacing: **Ratify fixed spacing, not overlap.** Pick one fixed spacing after the AABB audit using max post-scale width plus margin. Overlap should not be treated as acceptable diagnostic output because it damages side-by-side comparison.
- Q8 keyboard shortcuts: **Defer.** Optional polish only; not part of acceptance.

## Verdict

**CHANGES_REQUESTED.** The plan is aligned with the North Star, but the substrate and validation ambiguities above should be resolved in the spec before WP-A starts. After those amendments, proceed with WP-A scale calibration first, then WP-B Showroom implementation.
